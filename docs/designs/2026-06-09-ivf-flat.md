# IVF_FLAT Design

## Problem Statement

The current index is IVF-PQ. Query execution first selects `nprobe` IVF lists and
then ranks candidates with PQ-code distances. Low recall can come from two
independent sources:

1. IVF miss: a true neighbor is assigned to a list that is not probed.
2. PQ error: a true neighbor is in a probed list but loses during approximate
   PQ-distance ranking.

LanceDB's `IVF_HNSW_FLAT` points at the same direction: keep raw vectors in IVF
partitions to avoid quantization loss, then optionally accelerate per-partition
search with HNSW. This first change implements the raw-vector `IVF_FLAT`
baseline only. HNSW acceleration should follow after the FLAT path is merged and
benchmarked on production-shaped data.

## Chosen Approach

Add a new high-recall index family instead of changing the existing IVF-PQ
format:

1. Build `IVF_FLAT` first to isolate recall loss.
2. Keep IVF-PQ as the compact default path.
3. Defer `IVF_HNSW_FLAT` until HNSW can match IVF-FLAT recall closely enough.

This keeps the rollout small and gives us measurable evidence before paying the
cost of graph storage and graph-search complexity.

## Design Details

### Index Families

`IVFPQIndex` remains unchanged and continues to own:

- IVF centroids
- PQ codebooks
- per-list ids
- per-list PQ codes

Add a separate `IVFFlatIndex`:

- `d`
- `nlist`
- `metric`
- `quantizer_centroids`
- `ids: Vec<Vec<i64>>`
- `vectors: Vec<Vec<f32>>`, row-major per list

### Query Flow

`IVF_FLAT`:

1. Preprocess the query.
2. Select the nearest IVF lists with `kmeans::find_topk`.
3. Exact-scan raw vectors inside those lists.
4. Merge candidates into a global top-k heap.

### Metrics

Support the same public metric names as IVF-PQ:

- L2: rank by squared L2.
- Inner product: rank by negative inner product internally.
- Cosine: normalize vectors at train/add/search time and use L2 or cosine
  distance consistently.

First implementation should avoid OPQ. OPQ is useful for quantization but does
not naturally belong in a raw-vector FLAT index. It can be revisited only if a
caller needs rotated-space search compatibility.

### Filtering

Keep the existing `RowIdFilter` abstraction. For `IVF_FLAT`, filtering is cheap
and exact because every vector in a probed list is visited. The reader also
accepts serialized `RoaringTreemap` bytes so Java and Python callers can reuse
the existing filter exchange pattern.

### File Format

Do not extend the existing `IVPQ` file format in place. It is tightly shaped
around PQ code storage and delta-varint ids.

Add a new magic/version for raw-vector indexes:

- `IVFL` for `IVF_FLAT`

The `IVF_FLAT` layout should mirror the existing reader-friendly layout:

- header
- optional flags
- centroids
- offset table: list offset, list count, encoded id bytes length
- list payload: base id, delta-varint ids, raw `f32` vectors

### Build Strategy

Start single-node/in-memory, matching current `IVFPQIndex::add` behavior.
Distributed build can follow the existing `from_trained` pattern:

1. Train centroids once.
2. Workers assign rows into lists and emit raw-vector partitions.
3. Merge partitions by list.

### Public API

First expose Rust core APIs only:

- `IVFFlatIndex::new`
- `train`
- `add`
- `search`
- `write_ivfflat_index`
- `IVFFlatIndexReader::open`
- `search_with_filter`
- `search_batch_ivfflat_reader`

The first binding layer exposes:

- Java `IVFFlatWriter` / `IVFFlatReader`.
- Python `IVFFlatWriter` / `IVFFlatReader`.

Avoid changing existing `IVFPQReader`/`IVFPQWriter` semantics.

## Success Criteria

Benchmark on the same synthetic and real-like datasets:

- brute force recall baseline
- current IVF-PQ recall and latency
- IVF-FLAT recall and latency

Decision gates:

- If IVF-FLAT recall is still low, the primary problem is IVF partitioning or
  `nprobe`, not PQ. Tune `nlist`, `target_partition_size`, and `nprobe` first.
- If IVF-FLAT recall is high but slow, proceed to a separate HNSW acceleration
  proposal.

## Current Prototype Findings

The prototype changes the recommendation order:

- `IVF_FLAT` immediately diagnoses the recall issue. On the synthetic benchmark,
  IVF-PQ stays around 49% recall@10 even when probing all lists, while IVF-FLAT
  reaches almost full recall with modest `nprobe`.
- This strongly suggests the current low recall is mainly PQ quantization /
  approximate ranking error, not IVF partition miss.

Practical next step: merge `IVF_FLAT` as the high-confidence recall baseline and
optimize HNSW graph construction separately.

## Out of Scope

- Replacing IVF-PQ as the default index.
- Supporting OPQ in FLAT indexes.
- Implementing `IVF_HNSW_FLAT`.
- Implementing IVF_HNSW_PQ or IVF_HNSW_SQ.
- Online incremental graph updates.
- Cross-list graph links.

## Open Questions

- What recall target should we optimize for: Recall@10, Recall@100, or query
  layer top-k?
- What production dimension and row-count ranges should drive default
  `target_partition_size`?
- Should Java/Python expose separate classes or a generic vector-index factory?
